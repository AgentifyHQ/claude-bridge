# claude-bridge

A bridge for running Claude Code on remote servers via SSH — synchronous or async, with session continuity and file transfer.

## What it does

```
Local Machine                          Remote Server
┌─────────────────┐                    ┌─────────────────┐
│ Claude Code     │   SSH              │ Claude Code     │
│ (local)         │ ────────────────>  │ (remote)        │
│                 │                    │                 │
│ ask ────────────│──> claude -p ─────>│ runs task       │
│   <── stdout ───│<── result ────────<│ reports back    │
│                 │                    │                 │
│ send ───────────│──> inbox/*.json ──>│ process_tasks.py│
│ results <───────│<── outbox/*.json <─│ (async worker)  │
│                 │                    │                 │
│ push/pull ──────│──> scp ──────────> │ file transfer   │
└─────────────────┘                    └─────────────────┘
```

**Features:**
- **Sync mode** (`ask`) — ask remote Claude, get response immediately
- **Async mode** (`send`/`process`) — queue tasks, process later, check results
- **Session continuity** — multi-turn conversations via `--continue`
- **File transfer** — `push`/`pull` files between local and remote
- **Reporting** — remote Claude provides structured reports of what it did
- **Multi-server** — set up bridges to multiple servers from different projects
- Three interfaces: `bridge.sh` (zero deps), `justfile`, or Claude Code via ssh-mcp

## How it works

Run `setup.sh` from your **project folder**. It:
1. Installs Claude Code on the remote server
2. Deploys bridge scripts (task processor, worker, submitter) to the remote
3. Generates `.claude-bridge/` in your current directory with a `bridge.sh` CLI

```
your-project/
├── .claude-bridge/       # generated by setup.sh (gitignore this)
│   ├── bridge.sh         # CLI — all commands go through here
│   └── .mcp.json         # ssh-mcp config for Claude Code
├── .mcp.json             # copy from .claude-bridge/ for Claude Code
├── justfile              # your project justfile (optional, wraps bridge.sh)
└── ...
```

On the remote:
```
~/claude-bridge/          # bridge infrastructure
├── inbox/                # async task queue
├── outbox/               # async results
├── process_tasks.py      # task processor
├── bridge-worker.sh      # background watcher
└── submit_task.py        # task submitter

~/claude-workspace/       # remote Claude's working directory
└── CLAUDE.md             # agent context + reporting guidelines
```

## Quick start

```bash
# 1. Clone claude-bridge somewhere (or just have setup.sh accessible)
git clone https://github.com/AgentifyHQ/claude-bridge.git ~/claude-bridge

# 2. Go to your project folder
cd ~/my-project

# 3. Run setup — deploys to remote, generates .claude-bridge/ here
~/claude-bridge/setup.sh user@hostname

# 4. Authenticate Claude Code on the remote (one-time)
ssh user@hostname
claude  # follow login prompts, then exit

# 5. Start using it
.claude-bridge/bridge.sh ask "check disk usage and report"
.claude-bridge/bridge.sh ask -c "what about memory?"
```

## Setup options

```
setup.sh user@host [options]

Options:
  --key PATH       SSH key path (default: ~/.ssh/id_ed25519)
  --port PORT      SSH port (default: 22)
  --name NAME      Server name for config (default: derived from host)
  --bridge-dir DIR Remote bridge directory (default: ~/claude-bridge)
  --workspace DIR  Remote workspace directory (default: ~/claude-workspace)
  --no-install     Skip Claude Code installation on remote
```

## Usage

### bridge.sh (zero dependencies — just bash + ssh)

**Synchronous (ask and wait):**

| Command | Description |
|---------|-------------|
| `.claude-bridge/bridge.sh ask "prompt"` | Ask remote Claude, get response immediately |
| `.claude-bridge/bridge.sh ask -c "prompt"` | Follow-up (continues previous session) |

**Async (queue and process):**

| Command | Description |
|---------|-------------|
| `.claude-bridge/bridge.sh send "prompt"` | Queue a new task |
| `.claude-bridge/bridge.sh follow-up "prompt"` | Queue a follow-up (continues session) |
| `.claude-bridge/bridge.sh process` | Process all pending tasks |
| `.claude-bridge/bridge.sh results` | Show all completed results |
| `.claude-bridge/bridge.sh read <task-id>` | Read a specific result |
| `.claude-bridge/bridge.sh clear` | Clear all results |

**File transfer:**

| Command | Description |
|---------|-------------|
| `.claude-bridge/bridge.sh pull <remote> <local>` | Download file from remote |
| `.claude-bridge/bridge.sh push <local> <remote>` | Upload file to remote |

**Worker and server management:**

| Command | Description |
|---------|-------------|
| `.claude-bridge/bridge.sh worker-start` | Start background worker |
| `.claude-bridge/bridge.sh worker-stop` | Stop background worker |
| `.claude-bridge/bridge.sh worker-status` | Check worker and queue status |
| `.claude-bridge/bridge.sh ssh "cmd"` | Run any command on remote |

### With a justfile

Add a justfile to your project that wraps bridge.sh:

```just
bridge := justfile_directory() / ".claude-bridge/bridge.sh"

ask prompt:
    {{bridge}} ask "{{prompt}}"

follow-up prompt:
    {{bridge}} ask -c "{{prompt}}"

send prompt:
    {{bridge}} send "{{prompt}}"

process:
    {{bridge}} process

results:
    {{bridge}} results

ssh-cmd cmd:
    {{bridge}} ssh "{{cmd}}"
```

Then: `just ask "check disk usage"`

### With Claude Code (ssh-mcp)

Copy the generated `.mcp.json` to your project root and restart Claude Code:

```bash
cp .claude-bridge/.mcp.json .
```

This gives your local Claude direct SSH access to the remote server via MCP tools (`exec` and `sudo-exec`).

## Multi-server setup

Run setup from different project folders, or multiple times from the same folder with different names:

```bash
# From project-a/
~/claude-bridge/setup.sh admin@10.0.1.50 --name file-server

# From project-b/
~/claude-bridge/setup.sh deploy@prod.example.com --port 2222

# From homelab/
~/claude-bridge/setup.sh pi@192.168.1.100
```

Each project gets its own `.claude-bridge/` with server-specific config.

To give Claude Code access to multiple servers in one project, merge `.mcp.json` files manually or use separate MCP server names.

## Reporting

The remote Claude agent is instructed (via CLAUDE.md) to provide structured reports:

1. **What it did** — commands run, files modified, actions taken
2. **What it found** — results, status, errors, data
3. **What needs attention** — warnings, recommendations, follow-up actions

This means you get actionable output, not just raw command dumps.

## Post-setup customization

After running `setup.sh`, you can customize per-server:

### Add server-specific tools

Edit `.claude-bridge/bridge.sh` and add tools to the `ALLOWED_TOOLS` line:

```bash
ALLOWED_TOOLS="Bash(mycli:*) Bash(kubectl:*) Bash(ls:*) ..."
```

Also update `~/claude-bridge/process_tasks.py` on the remote for async mode.

### Add server context

Edit `~/claude-workspace/CLAUDE.md` on the remote to give the bridge agent server-specific context (installed services, CLI tools, project info, etc.).

### Environment variables (on remote)

| Variable | Default | Description |
|----------|---------|-------------|
| `CLAUDE_BRIDGE_DIR` | `~/claude-bridge` | Bridge directory |
| `CLAUDE_BRIDGE_WORKSPACE` | `~/claude-workspace` | Working directory |
| `CLAUDE_BRIDGE_BIN` | `~/.local/bin/claude` | Claude binary path |
| `CLAUDE_BRIDGE_SESSION` | `claude-bridge` | Session name |
| `CLAUDE_BRIDGE_ALLOWED_TOOLS` | (see source) | Allowed tools |

## Local cross-folder communication

For local-to-local communication (folder A to folder B on the same machine), you don't need claude-bridge. Claude Code has this built in:

**Using the Agent tool:**
```
# In your Claude Code session, launch a subagent in another directory
Agent: "Go to /path/to/other-project and check the test results"
```

**Using claude -p directly:**
```bash
cd /path/to/folder-b && claude -p "check test results" --add-dir /path/to/folder-a
```

**Using --add-dir in session:**
```bash
claude --add-dir /path/to/folder-b
# Now Claude can read/write both directories
```

The Agent tool spawns a fresh Claude instance as a subprocess — effectively a local bridge without any infrastructure. Use claude-bridge when you need to cross machine boundaries over SSH.

## Requirements

**Local:** `bash`, `ssh` with key auth.

**Remote:** Python 3.8+, Node.js (for Claude Code install only).
